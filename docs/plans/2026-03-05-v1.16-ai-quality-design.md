# v1.16 — AI Response Quality Design

**Date:** 2026-03-05
**Status:** Approved

## Problem Statement

User feedback identified two primary failure modes in AI responses:

1. **Wrong aggregation / hallucinated numbers** — Claude misreads the `summaryStats` object (e.g. cites the max value instead of the median, or references the wrong time period). The chart shows correct data but the analysis text contradicts it.
2. **Poor analysis quality** — Responses are generic walls of prose with no structure, no recommendations, and no investor perspective. Users expect a BayutGPT-style analyst voice with clear sections, specific numbers, and actionable opinions.

Secondary issues (also addressed):
- Every query always shows a chart, even when the question is open-ended or investment-focused
- No way to switch between chart types within a post

## Approach

**Targeted surgery (Approach B):** Fix the data passed to Claude, rewrite the full-mode prompt, enrich the intent object, and add a chart tab switcher. Short mode and the reply flow are unchanged.

---

## Section 1 — Fix Hallucination: Richer `summaryStats`

### Current state
`summaryStats` contains only: `{ min, max, median, latestValue, firstValue }`. Claude has to infer everything else, leading to aggregation errors.

### New structure

```js
summaryStats: {
  // Full series as a labeled table — every data point Claude can reference
  series: [
    { month: "2023-01", value: 1_250_000, label: "Jan 2023" },
    { month: "2024-01", value: 1_480_000, label: "Jan 2024" },
    // ...one entry per data point
  ],

  // Pre-computed metrics — Claude only quotes these, never calculates
  metrics: {
    overallChange:      "+18.4%",
    overallChangeAbs:   "+AED 230,000",
    yoyChange:          "+12.1%",
    cagr:               "+6.8% CAGR",
    peakMonth:          "Mar 2024",
    peakValue:          "AED 1,620,000",
    troughMonth:        "Aug 2022",
    troughValue:        "AED 980,000",
    latestValue:        "AED 1,480,000",
    latestMonth:        "Jan 2024",
    totalTransactions:  4821,           // volume queries
    avgVolumePerMonth:  401,
  },

  // For comparison queries: one metrics block per series name
  seriesMetrics: {
    "Noya":  { overallChange: "+22%", latestValue: "AED 1.6M", ... },
    "Emaar": { overallChange: "+11%", latestValue: "AED 1.1M", ... },
  }
}
```

### Grounding clause (reinforced)

> CRITICAL: Every AED figure, percentage, transaction count, and date you write must appear verbatim in the SERIES or METRICS sections below. Do not calculate, estimate, or draw on training knowledge. If a number is not listed, do not mention it.

### Implementation
- `computeSummaryStats()` in `/src/utils/computeSummaryStats.js` is rewritten
- Handles single-series (trend) and multi-series (comparison) chart data
- Pre-computes all metrics with formatted strings (Claude copies, never formats)

## Section 2 — Adaptive Full-Mode Response Format

### Current state
Full-mode prompt instructs: "NO headers, NO bullets, NO markdown — 2-3 paragraphs of flowing prose." This produces unstructured, generic text.

### New approach
Full mode returns **structured JSON with markdown sections**. The format adapts based on `adaptiveFormat` (set by Claude during intent extraction — see Section 3).

Short mode is **unchanged**: still streams a single sentence of plain prose.

### Response schemas by format

**`trend`** (price trend, rate trend, volume trend):
```json
{
  "headline": "+18.4% over 24 months — Yas Island outpacing Abu Dhabi average",
  "keyMetrics": [
    { "label": "Overall change",  "value": "+AED 230,000 (+18.4%)" },
    { "label": "Latest median",   "value": "AED 1,480,000 (Jan 2024)" },
    { "label": "Peak",            "value": "AED 1,620,000 (Mar 2024)" },
    { "label": "YoY growth",      "value": "+12.1%" }
  ],
  "analysis": "markdown — 2 short paragraphs of trend narrative",
  "recommendation": "markdown — fully opinionated investor take"
}
```

**`comparison`** (project vs project, district vs district, layout breakdown):
```json
{
  "headline": "Noya leads with +22% — 2× faster than Emaar beachfront",
  "ranking": [
    { "rank": 1, "name": "Noya",  "metric": "+22%", "note": "strongest momentum" },
    { "rank": 2, "name": "Emaar", "metric": "+11%", "note": "stable but slower" }
  ],
  "analysis": "markdown — comparative narrative",
  "recommendation": "markdown — which option Claude recommends and why"
}
```

**`investment`** (open-ended investment/ROI questions):
```json
{
  "headline": "Yas Island shows early-stage appreciation signals",
  "summary": "One sentence direct answer to the question",
  "marketData": "markdown — what the data actually shows",
  "riskFactors": "markdown — honest caveats from the data",
  "recommendation": "markdown — direct investor opinion"
}
```

**`factual`** (simple count/fact questions):
```json
{
  "headline": "4,821 transactions recorded in 2024",
  "answer": "Direct plain-text answer",
  "context": "markdown — 1 paragraph of context (optional, omit if unnecessary)"
}
```

### Analyst voice
All responses use a fully opinionated analyst voice: genuine market opinions based strictly on the data (e.g. "this area is undervalued relative to supply constraints"). No hedge language like "it's worth noting that..."

## Section 3 — Intent Enrichment: `adaptiveFormat`, `chartNeeded`, `suggestedCharts`

### New fields added to `/api/analyze` intent response

```json
{
  "queryType": "price_trend",
  "chartType": "line",
  "title": "Yas Island median prices — 2022 to 2024",

  "adaptiveFormat": "trend",
  "chartNeeded": true,
  "suggestedCharts": ["line", "bar", "volume"]
}
```

**`adaptiveFormat`** — drives full-mode response flavor:
| Value | When |
|---|---|
| `"trend"` | `price_trend`, `rate_trend`, `volume_trend` |
| `"comparison"` | `project_comparison`, `district_comparison`, `layout_distribution` |
| `"investment"` | Open-ended investment/ROI questions with no specific data request |
| `"factual"` | Simple factual questions with a single numeric answer |

**`chartNeeded`** — controls auto-chart display:
- `true` → run DuckDB query immediately, show chart (existing behavior)
- `false` → no auto-chart; show analysis + chart switcher chips

Rules for `chartNeeded: false`:
- `adaptiveFormat` is `"investment"` or `"factual"` AND no specific project/district/date range was identified
- System cannot map the query to a known `queryType`

**`suggestedCharts`** — 2–3 chip labels Claude recommends. Each maps to a known `queryType` so the frontend knows exactly which DuckDB query to run:
- `"line"` → same query, line chart variant
- `"bar"` → same query, bar chart variant
- `"volume"` → `volume_trend` query for the same filters
- `"price_trend"` → `price_trend` query
- `"rate_trend"` → `rate_trend` query
- `""` → no chips (simple factual with obvious answer)

### System prompt update
The `/api/analyze` system prompt schema is extended to include these three fields with the above rules.

## Section 4 — Chart Tab Switcher UI

### Component: `ChartSwitcher`

Replaces the current static `DynamicChart` wrapper inside `PostCard`. Holds `activeChip` state locally — no global state.

### Behavior when `chartNeeded: true`

1. Primary chart auto-loads (unchanged UX for data queries)
2. Below the chart, chips appear for `intent.suggestedCharts`
3. Active chip is highlighted (filled, accent color)
4. Tapping a different chip re-runs the DuckDB query with the new chart type — **no API call**, local only
5. Chart swaps with a smooth transition

### Behavior when `chartNeeded: false`

1. No chart shown by default
2. Chips appear below the analysis text (none selected)
3. Tapping a chip runs the DuckDB query and slides the chart in above the chips
4. Tapping the active chip again collapses the chart (toggle)

### Visual layout

```
chartNeeded: true
┌─────────────────────────────────────────────┐
│  [Active chart]                             │
│  ● Line trend  ○ Bar chart  ○ Volume        │
└─────────────────────────────────────────────┘

chartNeeded: false (no chip selected)
┌─────────────────────────────────────────────┐
│  [AnalysisBlock — text + recommendation]    │
│  ○ Price trend  ○ Volume  ○ AED/sqm         │
└─────────────────────────────────────────────┘

chartNeeded: false (chip selected)
┌─────────────────────────────────────────────┐
│  [Chart slides in]                          │
│  ● Price trend  ○ Volume  ○ AED/sqm         │
│  [AnalysisBlock — text + recommendation]    │
└─────────────────────────────────────────────┘
```

### Implementation notes
- `ChartSwitcher` is a new component: `src/components/ChartSwitcher.jsx`
- Chip → queryType mapping lives in `src/utils/chipToQuery.js`
- Uses `transition-all duration-300` for chart slide-in/out
- On mobile: chips scroll horizontally if they overflow

## Section 5 — Frontend Rendering: `AnalysisBlock`

### Component: `AnalysisBlock`

New component (`src/components/AnalysisBlock.jsx`) replacing the plain `<p>` tag that currently renders `post.analysisText`.

### Streaming behaviour

**Short mode** (unchanged): streams plain text → renders as `<p>` during and after streaming.

**Full mode** (new): streams the complete JSON string. During streaming, a pulsing skeleton is shown. When streaming ends, the string is JSON-parsed and handed to `AnalysisBlock`.

### Render variants

**`trend`**: headline (large, bold) → keyMetrics row (3–4 pill chips) → `react-markdown` analysis → accent recommendation block

**`comparison`**: headline → ranked list rows (rank medal + name + metric + note) → `react-markdown` analysis → accent recommendation block

**`investment`**: headline → summary sentence → `react-markdown` marketData section → `react-markdown` riskFactors section → accent recommendation block

**`factual`**: headline (large number emphasis) → plain answer text → optional `react-markdown` context

### Markdown library
`react-markdown` + `rehype-sanitize` (~15 KB gzipped). Used only in full-mode prose sections. Does not affect short-mode post performance.

### Backward compatibility
`AnalysisBlock` checks: if `post.analysisText` is not valid JSON, render as a plain `<p>`. All existing posts in localStorage continue to work.
